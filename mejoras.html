Bug #1 · classification_utils.py : 46

Condición lógica muerta

En classify_transaction(), la rama sin_receptor nunca puede ejecutarse con la condición que tiene porque la primera parte es siempre True en ese punto del flujo.

# Si emisor == cliente → ya retornó en línea 36 if emisor_ced == client_ced: # línea 36 return "ingreso" if receptor_ced == client_ced: # línea 40 return "egreso" # ↓ Aquí `not emisor_ced == client_ced` es SIEMPRE True if not emisor_ced == client_ced and receptor_empty: return "sin_receptor" # ───────────────────────────────────── # Correcto: if receptor_empty: return "sin_receptor"

¿Por qué importa?

Aunque el resultado es el mismo, el código es engañoso para el siguiente desarrollador. Si alguien reordena las ramas o refactoriza, puede romper la lógica pensando que la condición extra era necesaria.

Contexto

Esta función clasifica cada factura como ingreso/egreso/sin_receptor/ors según la perspectiva del cliente. Es llamada por cada factura en cada pestaña, así que también tiene impacto en rendimiento si se expande la lógica.

Bug #3 · pdf_cache.py : 94

Cache JSON no atómico

La ironía: el sistema tiene un protocolo SHA-256 ultra-seguro para mover PDFs, pero el archivo de caché que acelera toda la carga puede corromperse con un corte de luz.

# Actualmente — NO atómico def save_cache(self): with open(self.cache_file, "w") as f: json.dump(self.cache, f) # ← si muere aquí: JSON roto # ───────────────────────────────────── # Correcto — escritura atómica def save_cache(self): tmp = self.cache_file.with_suffix(".tmp") with open(tmp, "w") as f: json.dump(self.cache, f) tmp.replace(self.cache_file) # ← atómico en Windows/Linux

¿Qué pasa si se corrompe?

Al iniciar la app, _load_cache falla silenciosamente y devuelve un dict vacío. Resultado: todos los PDFs se re-escanean desde cero, lo que puede tomar varios minutos en clientes con cientos de facturas sobre red.

Escenarios reales de corrupción

  • ⚡ Corte de electricidad durante guardado
  • 🌐 Pérdida de conexión a la unidad Z: a mitad de escritura
  • 🔄 Cierre forzado de la app (tarea finalizada)

Rendimiento #1 · classification_utils.py : 137

get_tab_statistics es 8× más lento de lo necesario

Para calcular los contadores de cada pestaña, llama a filter_records_by_tab 8 veces — una por pestaña. Cada llamada itera todos los registros y llama a classify_transaction por cada uno.

# Actual: O(8 × n) — itera TODO 8 veces tabs = ["todas", "ingreso", "egreso", ...] for tab in tabs: filtered = filter_records_by_tab(records, tab, ...) # ↑ llama classify_transaction por cada record # Correcto: O(n) — un solo recorrido counters = {tab: {"count": 0, "clasificados": 0} for tab in tabs} for r in records: tab = classify_transaction(r, cedula) counters[tab]["count"] += 1 # ...

Impacto real

Esta función se llama en cada recarga de la lista (al clasificar una factura, al cambiar de pestaña, al refrescar). Con 500 facturas:


• Actual: ~4 000 llamadas a classify_transaction

• Corregido: ~500 llamadas a classify_transaction

Contexto

La función existe porque las pestañas necesitan saber cuántas facturas hay en cada categoría para mostrar los badges con números. Es un cálculo legítimo, solo mal implementado.

Rendimiento #2 · classifier.py : 156

SQLite: nueva conexión por operación

# Actual — cada método abre y cierra def get_estado(self, clave): with sqlite3.connect(self.path) as conn: # ← ... def upsert(self, **kwargs): with sqlite3.connect(self.path) as conn: # ← ... def get_records_map(self): with sqlite3.connect(self.path) as conn: # ← ... # Mejor — WAL mode + conexión persistente def _ensure(self): self._conn = sqlite3.connect(self.path, check_same_thread=False) self._conn.execute("PRAGMA journal_mode=WAL")

Overhead en clasificación masiva

Al clasificar 50 facturas en lote, se abren y cierran 50 conexiones SQLite sobre una ruta de red. Cada connect() sobre Z:/ puede tomar 10–50 ms.


Con conexión persistente + modo WAL: 1 conexión, múltiples escrituras en una transacción.

Nota de seguridad

El threading.Lock() existente ya protege el acceso concurrente. Con check_same_thread=False más el Lock, la seguridad se mantiene igual. WAL permite lectura simultánea durante escritura.

Rendimiento #3 · pdf_cache.py vs classifier.py

Tamaño de chunk inconsistente en SHA-256

# pdf_cache.py — chunks de 8 KB def _compute_checksum(pdf_file, chunk_size=8192): h = hashlib.sha256() with open(pdf_file, "rb") as f: for chunk in iter(lambda: f.read(8192), b""): h.update(chunk) # classifier.py — chunks de 1 MB (✓ correcto) def sha256_file(path): digest = hashlib.sha256() with path.open("rb") as fh: for chunk in iter(lambda: fh.read(1024 * 1024), b""): digest.update(chunk)

Impacto en red

Un PDF de 2 MB con chunks de 8 KB requiere 256 llamadas al sistema de lectura. Con 1 MB, son solo 2 llamadas. Sobre una unidad de red (Z:/), cada llamada tiene latencia adicional.

Solución

Unificar ambas funciones. PDFCacheManager._compute_checksum debería usar sha256_file de classifier.py directamente, eliminando también la duplicación de lógica.

También: dos funciones con el mismo propósito

sha256_file(path) y _compute_checksum(pdf_file) hacen exactamente lo mismo. Innecesario mantener dos implementaciones.

Arquitectura #3 · Raíz del repo

Archivo huérfano fuera del scope

recover_clasificaciones.py

Archivo sin versionar (?? en git status) en la raíz del repositorio. Viola la regla de scope: todo debe estar dentro de gestor_contable/.


Posibles causas:


  • 📌 Script de recuperación de emergencia creado manualmente
  • 📌 Prototipo de funcionalidad que nunca se integró
  • 📌 Herramienta de diagnóstico temporal olvidada

Riesgos

• Si accidentalmente se hace git add . y el script tiene credenciales o rutas hardcodeadas, entra al historial público.


• Si el script modifica SQLite o mueve archivos, puede interferir con la app.

Acción recomendada

1. Revisar el contenido del script


2. Si es útil → moverlo a gestor_contable/tools/ y versionarlo


3. Si es temporal → eliminarlo y agregar *.py a .gitignore solo en raíz

UX #3 · Corrección de errores

Sin botón "Deshacer clasificación"

Flujo actual para corregir un error

1
Darse cuenta del error
La factura ya fue movida a la carpeta incorrecta
2
Ir a pestaña "Clasificadas"
Buscar la factura entre todas las clasificadas
3
Reclasificar manualmente
Seleccionar la categoría correcta otra vez
4
Esperar que se mueva de nuevo
Otra operación atómica SHA-256

Con Deshacer

1
Darse cuenta del error
Inmediatamente después de clasificar
2
Ctrl+Z o botón "Deshacer"
La factura vuelve a su posición original

Implementación

La información ya está en SQLite (ruta_origen, sha256). Se necesita una pila de "últimas N operaciones" en memoria y un botón/atajo que revierta la última, aplicando el mismo protocolo atómico en sentido inverso.
